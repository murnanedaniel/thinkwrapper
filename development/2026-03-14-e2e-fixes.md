# Session Log: End-to-End Loop Fixes — 2026-03-14

## Goal
Close the full user journey: topic → preview → Google OAuth → Paddle payment → DB entry → email delivered.

## Bugs Found & Fixed

### 1. Celery pool deadlock with Claude Agent SDK
**File:** `app/celery_config.py`
**Problem:** Celery's default `prefork` pool spawns child processes. The Claude Agent SDK uses `asyncio.run()` inside `generate_newsletter_sync()`, which deadlocks in forked processes.
**Fix:** Set `worker_pool='solo'` in celery config so workers run in the main process thread.

### 2. Claude Agent SDK blocked by nested session detection
**File:** `app/agent_service.py`
**Problem:** The SDK spawns a bundled Claude Code CLI subprocess. When running inside an existing Claude Code session, the `CLAUDECODE` env var triggers: *"Claude Code cannot be launched inside another Claude Code session."*
**Fix:** Pass `env={"CLAUDECODE": ""}` in `ClaudeAgentOptions` to suppress the check.

### 3. SDK using API key instead of Claude Code subscription
**File:** `app/agent_service.py`
**Problem:** With `ANTHROPIC_API_KEY` set in the environment, the bundled CLI uses expensive API calls instead of the user's Claude Code subscription. Confirmed via `claude auth status` — with the key set: `apiKeySource: "ANTHROPIC_API_KEY"`, without: `authMethod: "claude.ai", subscriptionType: "pro"`.
**Fix:** Pass `env={"ANTHROPIC_API_KEY": ""}` in `ClaudeAgentOptions` (unless `USE_API_KEY` env var is set as an escape hatch for production).

### 4. Frontend crash: `setProgressMsg` undefined
**File:** `client/src/pages/CreateNewsletter.jsx`
**Problem:** `handleGeneratePreview()` called `setProgressMsg('Starting...')` but no such state variable existed. The `ReferenceError` was caught by the try/catch which displayed "Failed to generate preview" — masking the real issue (the task was actually submitted successfully).
**Fix:** Removed the call to the nonexistent function.

### 5. Frontend polling timeout too short
**File:** `client/src/components/NewsletterGenerator.jsx`
**Problem:** `maxAttempts = 120` (120 seconds). Agent SDK generation takes 75–210 seconds, so the UI timed out before the task completed.
**Fix:** Increased to `maxAttempts = 300`. Also added handling for `RETRY` state which was falling through silently.

### 6. Stale closure in Paddle checkout callback
**File:** `client/src/pages/CreateNewsletter.jsx`
**Problem:** The Paddle `eventCallback` was registered once during the first render, capturing `formData` and `preview` at their initial empty values. After OAuth redirect, `useEffect` restored these from `localStorage`, but the callback still held the stale empty references. Result: `POST /api/newsletters` sent `topic: ""` → 400 "Topic is required".
**Fix:** Added `useRef` for `formData` and `preview`, kept in sync via `useEffect`. The Paddle callback now reads from refs, which always reflect current state.

### 7. Database schema drift — missing columns
**Problem:** `newsletters` table was created before `description`, `style`, and `status` columns were added to the model. `POST /api/newsletters` → `OperationalError: table newsletters has no column named description`.
**Fix:** `ALTER TABLE` to add the three missing columns.

### 8. Flask not loading `.env` file
**File:** `app/__init__.py`
**Problem:** No `load_dotenv()` call anywhere in the Flask app. Mailjet API keys from `.env` were never loaded, so `send_email()` silently returned `False`.
**Fix:** Added `from dotenv import load_dotenv; load_dotenv()` at module level in `app/__init__.py`.

### 9. Mailjet sender not verified
**Problem:** `danieltmurnane@gmail.com` was registered as a Mailjet sender but had Status: "Inactive" (unverified). Mailjet accepted API calls (200) but silently dropped the emails.
**Fix:** Triggered verification via Mailjet API (`/sender/{id}/validate`), user clicked confirmation link. Confirmed delivery by checking message status via API — changed from "dropped" to "queued" → delivered.
